Interactive Fiction — Architecture Overview

An Infocom homage built as a showcase of agentic AI patterns. Python + Textual + OpenRouter.
Overview Room Model Generation Verification Play Topology Evaluation & Observability Pattern Glossary Open Decisions

§1Overview

The player picks a theme (fantasy, space, noir, etc.) and a size (small / large / xlarge). Generation agents build a world. Verification agents prove it is solvable and coherent. Play agents orchestrate a living turn-by-turn game. Save files preserve full state; templates preserve the world skeleton for replays.

Core stance Agents sit on top of a deterministic substrate, not in place of it. Every room is a state machine. LLM calls produce prose, make judgments, and drive non-player characters — but the mechanics of the game are verifiable Python.

Decisions already locked

  • Interface: Python Textual TUI, distributed as uvx if_fun. Potential Rust port once the agent guts are cemented.
  • Input loop: hybrid parser + LLM-interpreter fallback. Canonical verbs parsed deterministically; unknowns routed to an Intent Interpreter.
  • Generation strategy: skeleton eager, prose lazy. Structural layer (map, puzzle DAG, story beats, placements) generated and verified up front; per-room prose is canonical and frozen on first generation.
  • Room model: rooms are state machines with an optional persona-agent overlay for ~5–15% of rooms (the memorable ones).
  • Selective elevation: same pattern applied to rooms, mobs (most reactive, named antagonists get planners), and hints. Scale complexity with narrative importance.
  • Verification: puzzle solvability proven by BFS over the composed state machine (model checking), not by LLM judgment. Editor revision loop capped at 3 iterations, fail loud on non-convergence.
  • Theme enforcement: threaded into every generator's system prompt, not a dedicated agent. Story Critic catches violations in verification.
  • Save format: single JSON file per save. Full snapshot every time.
  • Prompt caching: world skeleton is the stable cacheable prefix; turn-specific context appends after the cache breakpoint.
  • Testing: strict TDD for deterministic code; pydantic-contract + cassette tests for LLM code; live LLM eval suite excluded from pre-commit, run on CI / on-demand.
  • Observability: Langfuse for traces, costs, and historical score comparison. Prompts live in git; Langfuse receives traces only.

§2The Three-Layer Room Model

Every room has mechanical state and canonical prose. Only selected rooms get an agent persona. Conflating these is the architectural mistake we are explicitly avoiding.

optional · ~5–15% of rooms
Layer 3 — Persona
An LLM agent with voice, short-term memory, and optional autonomy. Narrates events in its room when invoked. Example: the Whispering Library that resents intruders and dims its lights when forbidden books are read.
frozen · always present
Layer 2 — Prose
Canonical description strings written once during generation and stored verbatim. LOOK returns this string byte-for-byte. Kills description drift before it starts.
deterministic · always present
Layer 1 — Mechanical
State variables, transitions, guards, effects. Pure Python, no LLM. Items present, occupants, flags (door_north: locked, chandelier: hanging). Serializable. Verifiable via model checking.
Why the split matters Drift prevention is a layer-2 problem (freeze canonical text). Solvability is a layer-1 problem (BFS the state machine). Personality is a layer-3 problem (agent overlays). Each concern lives in exactly one layer. This keeps 90% of rooms cheap and fast while the memorable 10% get the expensive agent treatment.

§3Phase 1 — Generation

Runs once when the player creates a new world. A sequential pipeline with a parallel fan-out at the prose stage.

§4Phase 2 — Verification

Every generated world goes through a battery of verifiers. Mechanical checks are deterministic code. Narrative checks are LLM critics. If any check fails, the Editor decides which generator to re-invoke and what feedback to give. Capped at three iterations.

AgentJobPatternModel
Map Verifier Checks graph connectivity, reachability from start, no orphaned rooms, geometry consistency (no A→B north but B→A east). deterministic —
Solvability Checker BFS over the world state machine. Proves the win state is reachable from the start state. Emits a winning action trace. model-checking —
Story Critic Reads all prose end-to-end. Flags tonal drift, contradictions, un-Chekhovian guns, loose ends. reflection Sonnet
Difficulty Judge Reads the solvability trace + puzzle DAG. Scores difficulty. Flags mismatches with the requested size tier. reflection Sonnet
Editor Reads all verifier reports. On failure, decides which upstream generator to re-invoke and with what feedback. Caps iterations. evaluator-optimizer Opus

§5Phase 3 — Play

Every turn: player input flows through a router (deterministic parser, with LLM fallback for novel phrasings) into the Game Master, which applies state-machine transitions and dispatches workers to produce the turn output.

AgentJobPatternModel
Parser Maps canonical verb-object inputs (go north, take lamp) to structured actions. Falls through on unknown. deterministic —
Intent Interpreter LLM fallback. Takes free-form player text + game context. Either normalizes to a canonical action or marks as creative / conversational. routing Haiku
Game Master Hot path (most turns): apply canonical state transition, dispatch narrator, return. Single LLM call or zero. Cold path (novel or free-form input): enter a ReAct loop — reason about intent, probe state via tools, propose transitions, observe legality, revise. Assembles final turn output either way. orchestrator-workers react tool-use Sonnet
Narrator Default voice. Given the mechanical delta + canonical prose + recent event log, produces the turn's prose output. worker Sonnet
Room Persona For sentient rooms only. GM hands off narration when the player is in/entering one. Autonomous variant (room has goals, like the Whispering Library wanting to keep secrets): runs a ReAct loop — reason about player behavior, pick an effect (dim lights, whisper, reveal a secret), observe, revise. handoff react Sonnet
Mob Agent Reactive tier (rats, goblins, patrol guards): single-shot per turn — perceive local state, pick an action, return. Planner tier (named antagonists, recurring NPCs): full ReAct loop with persistent multi-turn goals and a current plan; revises when the plan fails. hierarchical react tool-use Haiku / Sonnet
Memory Curator Prunes context, summarizes distant events, decides what to include in each LLM call. Prevents GM prompts from ballooning. context-engineering Haiku

§7Evaluation & Observability

Agentic systems drift silently. A disciplined eval framework + always-on tracing is how we catch regressions before users do. Every LLM call is traced to Langfuse; every LLM-backed agent has a pytest-driven eval suite.

What gets evaluated

AgentEval approachScoring
Map Designer Generate → run Map Verifier on output → check graph properties (connectivity, no one-way orphans, exit count distribution). Deterministic
Puzzle Designer Generate → run Solvability Checker → assert win-state reachable, DAG depth matches size tier. Deterministic
Intent Interpreter Golden dataset of player utterances → expected canonical action (or "free-form"). Measure exact-match + near-miss. Deterministic
Narrator Dataset of (state delta, expected tone). LLM-judge scores: on-theme, descriptive, doesn't contradict canonical prose. LLM-judge
Room Persona Dataset of (persona spec, triggering event, context). LLM-judge scores: voice consistency, appropriateness, does not break game state. LLM-judge
Mob Agent Dataset of (mob state, local perception). Assert valid action chosen; LLM-judge scores believability for named antagonists. Mixed
Story Critic, Difficulty Judge, Editor Regression dataset of known-bad worlds with known defects. Assert critics flag the expected issues. Deterministic

§8Agentic Pattern Glossary

Definitions and canonical references for every pattern used in the architecture. The patterns Anthropic calls "building blocks" come from Building Effective Agents; others are standard in the broader literature.

Orienting distinction — workflows vs. agents Anthropic draws a useful line: workflows have topology you design in advance (a chain, a router, an orchestrator calling known workers). Agents have LLM-driven control flow — the model decides what to do next in a loop. Our Generation pipeline is a workflow. Our Planner Mobs and the GM's cold path are agents. Both are legitimate; workflows are more predictable and cheaper, agents handle open-ended tasks. Most production "agent systems" are workflows with agent-shaped subcomponents.

react

Reason + Act. An agent interleaves reasoning ("Thought: ...") with tool calls ("Action: ...") and observations in a loop, until the task is done. The foundational agentic-loop pattern from Yao et al. 2022.

Status note: the literal text-parsed Thought:/Action:/Observation: format is largely obsolete — native tool calling replaced it. What survives is the loop shape: reason → act → observe → repeat. Most modern "agents" (as distinct from workflows) are ReAct-shaped whether or not the paper is cited. Often composes underneath other named patterns (orchestrator, evaluator-optimizer).

ReAct paper Anthropic (agents vs workflows)

prompt-chain

A fixed sequence of LLM calls where each step's output feeds the next. Good for workflows that decompose into predictable stages (plan → expand → refine). Trade-off: lower latency sometimes for accuracy, since each step specializes.

Anthropic

routing

A classifier (deterministic or LLM) directs input to one of several downstream handlers. Lets you use cheap specialists for cheap cases and expensive ones where needed.

Anthropic

parallelization

Same task runs on multiple inputs concurrently (sectioning) or multiple perspectives run on the same input and results merge (voting). Reduces wall time and improves robustness.

Anthropic

orchestrator-workers

A central orchestrator LLM breaks tasks down, dispatches specialized workers, and synthesizes their outputs. The orchestrator decides dynamically which workers to invoke.

Anthropic

evaluator-optimizer

A generator produces output, an evaluator judges it, feedback loops to the generator until the output passes. Classic for quality-gated generation.

Anthropic

reflection

A critic agent (often the same model with a different prompt) reviews its own or another agent's output and emits structured feedback. Reflexion paper formalizes this with a memory of past critiques.

Reflexion (Shinn et al.)

tool-use

An LLM calls typed functions (tools) that read or mutate external state. The model's job becomes choosing which tool and what arguments. Every state mutation in our architecture goes through a tool.

Anthropic docs OpenAI

handoff

One agent transfers control (and context) to another, typically for a specialization reason. The first agent stops, the second takes over. Used when dispatch decisions happen at runtime based on context.

OpenAI Agents SDK

hierarchical

Agents nest: a parent orchestrator contains child agents that themselves may contain sub-agents. Mobs-under-GM is a one-level nesting. LangGraph's multi-agent docs treat this as a graph of graphs.

LangGraph

structured-output

LLM output is constrained to a typed schema (JSON / pydantic / zod). Makes downstream code reliable. pydantic-ai and instructor are the Python mainstays.

pydantic-ai instructor

model-checking

Not an agent pattern — a verification technique. Exhaustively explores a finite state space to prove a property (like "the win state is reachable"). We use BFS over the world state machine for puzzle solvability.

Wikipedia

context-engineering

The discipline of deciding what goes into each LLM call: what to cache, what to summarize, what to drop. Prompt caching, memory curation, and retrieval all fall here.

Anthropic

llm-as-judge

An LLM scores another LLM's output against criteria (coherence, tone, factuality). Used for subjective evaluation that deterministic checks can't cover. Best paired with reference outputs and a rubric to reduce variance.

MT-Bench paper HF cookbook

observability-as-code

Every LLM call is traced — inputs, outputs, latency, cost, tokens, model version — to a dedicated backend. Not an agent pattern, but a non-negotiable discipline for production agentic systems. Langfuse, LangSmith, Arize Phoenix, and Helicone are current options.

Langfuse

§9Open Decisions

Four things I'd like your call on before we lock the design and write the spec.

1 · Editor revision loop

Most agentically interesting piece, but also where spirals can happen. Keep the evaluator-optimizer loop (up to 3 iterations), or fail fast and regenerate from scratch on any verification failure?

2 · Mob cognition depth

Do mobs plan (think multi-turn ahead, form goals) or just react (decide this turn only)? Planning impresses; reactive is cheaper and less prone to weirdness. Could also vary per-mob.

3 · Hint Giver agent

Player types HINT and receives an escalating nudge sequence. Not in the current diagram. Want it in v1, or later?

4 · Theme authority

Who enforces "this is a space game, not a fantasy game"? A dedicated Theme Warden agent (more agentic), or a constraint threaded through every generator's system prompt (cheaper, simpler)?
